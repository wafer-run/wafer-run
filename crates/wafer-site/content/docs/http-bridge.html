HTTP Bridge

The @wafer/http-listener block converts incoming HTTP requests into wafer messages and dispatches them to flows or blocks. It is built on axum and lives in the wafer-core crate (enable the http feature). The runtime spawns the HTTP server automatically on start().

Setup

The easiest way to get an HTTP server with security headers, CORS, rate limiting, monitoring, and routing is @wafer/http-server:

[dependencies]
wafer-core = { version = "0.1", features = ["http"] }
let mut w = Wafer::new();
wafer_core::flows::http_server::register(&mut w, serde_json::json!({
    "listen": "0.0.0.0:8080",
    "routes": [{ "path": "/**", "block": "my-handler" }]
}));

let w = w.start().await.unwrap();
tokio::signal::ctrl_c().await.unwrap();
w.shutdown().await;

For advanced use, configure @wafer/http-listener directly with a custom flow or single block target:

w.add_block_config("@wafer/http-listener", serde_json::json!({
    "flow": "my-flow",
    "listen": "0.0.0.0:8080",
}));
// or dispatch to a single block:
w.add_block_config("@wafer/http-listener", serde_json::json!({
    "block": "@wafer/web",
    "listen": "0.0.0.0:8080",
}));

Method to Action Mapping

Each HTTP method is mapped to a semantic action stored in req.action:

  • GET / HEAD → retrieve
  • POST → create
  • PUT / PATCH → update
  • DELETE → delete
  • Any other method → execute

Meta Keys

The bridge populates these meta fields on each message:

  • http.method — the HTTP method (GET, POST, etc.)
  • http.path — the request path
  • http.raw_query — the raw query string
  • http.remote_addr — client IP (from X-Forwarded-For or socket)
  • http.content_type — request content type
  • http.host — the Host header
  • http.header.* — all request headers (e.g. http.header.authorization)
  • http.query.* — query parameters (e.g. http.query.page)

The bridge also sets normalized meta keys used by blocks:

  • req.action — semantic action (retrieve, create, update, delete)
  • req.resource — the request path
  • req.param.* — path parameters from {var} captures
  • req.query.* — query parameters (mirrors http.query.*)
  • req.client.ip — client IP address
  • req.content_type — request content type

Response Helpers

// Raw response with content type
respond(msg, data_bytes, "text/plain")

// JSON response
json_respond(msg, &json!({"status": "ok"}))

// Error helpers (transport layer maps error codes to HTTP status)
err_bad_request(msg, "Invalid input")
err_not_found(msg, "User not found")
err_unauthorized(msg, "Token expired")
err_internal(msg, "Something went wrong")

ResponseBuilder

For fine-grained control over the response (custom headers, cookies, explicit status):

new_response(msg)
    .set_header("X-Custom", "value")
    .set_cookie("session=abc123; HttpOnly; Secure")
    .json(&json!({"ok": true}))

Or with a raw body:

new_response(msg)
    .set_header("X-Custom", "value")
    .body(data_bytes, "application/octet-stream")

Use .status(code) to set an explicit HTTP status (e.g. for redirects):

new_response(msg).status(301)
    .set_header("Location", "/new-path")
    .body(b"Redirecting".to_vec(), "text/plain")

Built-in Blocks

wafer ships with ready-to-use blocks in the @wafer-run/wafer-run namespace.

wafer-run/http-server

Batteries-included HTTP server. Registers a flow that chains security-headers, CORS, readonly-guard, rate-limiting, monitoring, and the config-driven router — plus the HTTP listener that binds to a port. One call to set up a complete server:

wafer_core::flows::http_server::register(&mut w, json!({
    "listen": "0.0.0.0:8080",
    "routes": [
        { "path": "/api/users", "actions": ["retrieve"], "block": "list-users" },
        { "path": "/api/users/{id}", "block": "get-user" },
        { "path": "/**", "block": "wafer-run/web" }
    ]
}));

Config:

  • listen — the address to listen on (default: "0.0.0.0:8080")
  • routes — route array passed to wafer-run/router (see below)

The config is automatically split and distributed to the underlying blocks. You can still override individual block configs (e.g. wafer-run/cors) separately.

wafer-run/http-listener

Low-level HTTP transport block that listens for HTTP requests, converts them to standard WAFER messages, and converts results back to HTTP responses. Most users should use wafer-run/http-server instead. Configured as a block config (not a flow node) — it spawns its own TCP listener on startup.

Config:

  • flow — the flow ID to execute for each request (mutually exclusive with block)
  • block — a single block to dispatch to (mutually exclusive with flow)
  • listen — the address to listen on (e.g. "0.0.0.0:8090")
w.add_block_config("wafer-run/http-listener", json!({
    "flow": "my-flow",
    "listen": "0.0.0.0:8090"
}));

wafer-run/router

Config-driven router that matches incoming messages by action and path, then dispatches to handler blocks via ctx.call_block(). Returns 404 Not Found when no route matches. Transport-agnostic — works with any message that has standard meta properties.

Config:

  • routes — array of route objects, each with:
    • path — path pattern with {var} params and /** wildcard support
    • actions — optional array of actions (e.g. ["retrieve", "create"]); if omitted, matches any action. Also accepts methods with HTTP verbs (["GET", "POST"]) which are automatically mapped to actions.
    • block — the block name to call when matched

Action mapping: GET/HEAD → retrieve, POST → create, PUT/PATCH → update, DELETE → delete, OPTIONS → execute.

Path variables captured by {var} patterns are set as req.param.var in the message meta. Routes are evaluated in order — the first match wins.

{
    "block": "wafer-run/router",
    "config": {
        "routes": [
            { "path": "/users", "actions": ["retrieve"], "block": "my-app/list-users" },
            { "path": "/users/{id}", "actions": ["retrieve"], "block": "my-app/get-user" },
            { "path": "/users", "actions": ["create"], "block": "my-app/create-user" },
            { "path": "/health", "block": "my-app/health" }
        ]
    }
}

wafer-run/security-headers

Adds standard security headers to all responses: X-Content-Type-Options, X-Frame-Options, X-XSS-Protection, Referrer-Policy, Strict-Transport-Security, Content-Security-Policy, and Permissions-Policy.

Config: csp — override the default Content-Security-Policy value.

wafer-run/cors

Handles CORS preflight (OPTIONS) requests and injects Access-Control-* headers. Supports origin allowlisting, credentials, and configurable max-age.

Config:

  • allowed_origins — comma-separated origins or * (default: *)
  • allowed_methods — allowed HTTP methods (default: GET, POST, PUT, PATCH, DELETE, OPTIONS)
  • allowed_headers — allowed headers (default: Content-Type, Authorization, X-Requested-With)
  • max_age — preflight cache duration in seconds (default: 86400)

wafer-run/ip-rate-limit

Per-IP sliding-window rate limiting. Returns 429 Too Many Requests when the limit is exceeded and sets X-RateLimit-Limit, X-RateLimit-Remaining, and Retry-After headers.

Config:

  • max_requests — max requests per window (default: 1000)
  • window_seconds — window duration in seconds (default: 60)

wafer-run/readonly-guard

Rejects write operations (create, update, delete actions) with 403 Forbidden when read-only mode is enabled.

Config: readonly — set to "true" to enable (default: "false").

wafer-run/monitoring

Tracks request metrics: total count, error count, status code distribution, and path frequency. Exposes a stats endpoint at /_stats or /_monitoring that returns JSON with uptime and request data.

wafer-run/auth-validator

Authentication middleware supporting three token types:

  • JWT Bearer tokens — validated via the crypto service
  • API keys (sb_ prefix) — hashed with SHA-256 and checked against the database for revocation/expiration
  • httpOnly cookies — session-based authentication

On success, sets auth.user_id, auth.user_email, and auth.user_roles in the message meta for downstream blocks.

wafer-run/iam-guard

Role-based access control. Checks if the authenticated user has the required role. Returns 401 if not authenticated, 403 if the role is missing.

Config: role — required role name (default: "admin").

wafer-run/web

Static file server with MIME detection, SPA support, and intelligent caching. Blocks path traversal and dotfile access.

Config:

  • web_root — directory to serve from (default: "./public")
  • web_prefix — URL prefix to strip (default: empty)
  • web_spa — serve index.html for missing routes (default: "false")
  • web_index — index filename (default: "index.html")
  • cache_max_age — cache duration in seconds (default: 3600)
  • immutable_max_age — cache duration for hashed assets (default: 31536000)

Pre-built Flows

The wafer-run/http-server flow chains all infrastructure blocks (security-headers → cors → readonly-guard → ip-rate-limit → monitoring → router) and is the recommended starting point for HTTP servers. Register it with wafer_core::flows::http_server::register(&mut w, config).

For custom flows, compose blocks directly in your flow definition:

{
    "id": "my-flow",
    "root": {
        "block": "wafer-run/security-headers",
        "next": [{
            "block": "wafer-run/cors",
            "next": [{
                "block": "wafer-run/auth-validator",
                "next": [{
                    "block": "my-handler"
                }]
            }]
        }]
    }
}

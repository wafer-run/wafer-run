Package Registry

The package registry at https://wafer.run/registry is a discovery index for wafer blocks, flows, and interfaces. Block names use the {org}/{block} format (e.g. wafer-run/sqlite, acme/auth). Resolution is backed by a static registry repo at github.com/wafer-run/registry.

Block Naming

All block names follow the canonical {org}/{block} format — no @ prefix, exactly two segments separated by /. Examples:

  • wafer-run/sqlite — core SQLite block
  • acme/auth — third-party auth block
  • db, storage — aliases (any string without /)

Versioned references use @version suffix: acme/auth@v1.0.0. Omit the version to resolve latest.

Publishing a Block

Blocks can be published as WASM (cross-language, sandboxed) or native Rust crates (zero-overhead, full access). Both use the same shared types from wafer-block.

WASM blocks

Publishing is three steps:

  1. Build and release — push your block to GitHub, build the .wasm file, create a GitHub Release with the .wasm as a release asset
  2. Add a registry manifest — submit a PR to wafer-run/registry adding {org}/{block}/manifest.json
  3. Merge — once merged, the block is resolvable by the runtime
# Build your block as WASM
npx wafer-sdk-ts build -o myblock.wasm

# Tag and push
git tag v0.1.0
git push origin v0.1.0

# Then create a release on GitHub and upload the .wasm file
# Build your block as WASM
GOOS=wasip1 GOARCH=wasm go build -o myblock.wasm .

# Tag and push
git tag v0.1.0
git push origin v0.1.0

# Then create a release on GitHub and upload the .wasm file
# Build your block as WASM
cargo build --release --target wasm32-wasip1

# Tag and push
git tag v0.1.0
git push origin v0.1.0

# Then create a release on GitHub and upload the .wasm file

Registry manifest

Add a manifest to the wafer-run/registry repo at {org}/{block}/manifest.json:

{
    "name": "acme/auth",
    "summary": "JWT authentication block",
    "latest": "0.1.0",
    "versions": {
        "0.1.0": {
            "abi": 1,
            "wasm_url": "https://github.com/acme/auth-block/releases/download/v0.1.0/auth.wasm"
        }
    }
}

Fields:

  • abi — integer ABI version for WASM compatibility (currently 1)
  • wasm_url — URL to download the .wasm block (omit for native-only blocks)
  • flow_url — URL to download a .flow.json (for flow packages)
  • crate — optional Rust crate name for native compilation

Native Rust blocks

Native blocks are published as regular Rust crates. Add them as Cargo dependencies and register at startup:

# Cargo.toml
[dependencies]
wafer-run = "0.1"
my-auth-block = { git = "https://github.com/acme/auth-block", tag = "v1.0.0" }
// main.rs
use my_auth_block::AuthBlock;

let mut w = Wafer::new();
w.register_block("acme/auth", Arc::new(AuthBlock::new()));

Native blocks compile into your binary — no runtime download, no sandbox overhead. They have full access to the host and can use async Rust directly.

Publishing a Flow

Publishing a flow follows the same workflow:

  1. Write a .flow.json describing your flow
  2. Create a GitHub Release and attach the .flow.json as a release asset
  3. Add a registry manifest with a flow_url instead of wasm_url

A flow definition is a JSON file that wires blocks together. For example:

{
    "id": "acme/my-flow",
    "root": {
        "block": "acme/auth",
        "next": [
            { "match": "GET:/api/*", "block": "acme/api-handler" },
            { "match": "*", "block": "acme/static-server" }
        ]
    }
}

Publishing an Interface

Interface packages are reusable JSON Schema type definitions that blocks can reference for their input/output contracts. Publishing follows the same GitHub workflow:

  1. Push your interface definition to a public GitHub repository
  2. Write a .interface.json file describing the shared schema
  3. Create a GitHub Release and attach the .interface.json as a release asset

An interface definition is a JSON file with a name, version, description, and a JSON Schema. For example:

{
    "name": "http-request",
    "version": "1.0.0",
    "description": "Standard HTTP request shape",
    "schema": {
        "type": "object",
        "properties": {
            "method": { "type": "string" },
            "path": { "type": "string" },
            "headers": { "type": "object", "additionalProperties": { "type": "string" } },
            "body": { "type": "string" }
        },
        "required": ["method", "path"]
    }
}

Name the file {block}.interface.json (e.g. http-request.interface.json) and attach it to your GitHub Release.

Interfaces are opt-in — blocks can still use inline JSON Schema for their input/output definitions. Interfaces are useful when multiple blocks need to share the same type contract.

Resolution Order

When the runtime encounters a block name, it resolves in this order:

  1. Alias — resolve to target name, restart
  2. Already registered — use the native block (it called register())
  3. Registry — fetch manifest from raw.githubusercontent.com/wafer-run/registry/main/{org}/{block}/manifest.json, check ABI version, download wasm_url or flow_url

The runtime checks ABI compatibility before loading — if the manifest's abi field doesn't match the runtime's ABI version, it returns a clear error.

For native-only blocks (no wasm_url or flow_url), the runtime returns an error with the cargo add command to compile the block in.

Installing a Block

Reference a published block by name in your flow configuration — the runtime resolves it via the registry during startup:

{
    "root": {
        "block": "acme/auth",
        "next": [
            { "match": "GET:/api/data", "block": "my-handler" }
        ]
    }
}

Pin a version with @version for production: acme/auth@v1.0.0. Without a version, the registry's latest field is used.

Browsing and Searching

  • Browse the registry at https://wafer.run/registry
  • Search via API: GET https://wafer.run/registry/search?q=auth

Registry API

All endpoints are at https://wafer.run:

  • GET /registry/search?q=term&type=block|flow|interface — search indexed packages. The optional type parameter filters results by package type.
  • GET /registry/packages/{org}/{block} — package details + versions (fetched from the registry manifest)
  • GET /registry/packages/{org}/{block}/versions — version list from the registry manifest
  • GET /registry/packages/{org}/{block}/download/{version}?type=block|flow — redirects to the asset URL from the manifest

Removing a Package

Remove the manifest from the registry repo. If you also remove your GitHub repository or delete a release, the asset URLs will stop resolving.

Reporting a Bad Package

If you find a package that is malicious, contains malware, or otherwise violates the terms of use, please open a GitHub issue to request an admin review. Include the package name and a description of the problem.

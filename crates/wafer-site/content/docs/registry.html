Package Registry

The package registry at https://wafer.run/registry is a discovery index for wafer blocks, chains, and interfaces. Like Go modules, there is no registration step — just push your code to GitHub, tag a release, and the registry indexes it automatically the first time someone looks it up.

Publishing a Block

Publishing is three steps — all on GitHub, nothing to do on wafer.run:

  1. Push your block to a public GitHub repository
  2. Build the .wasm file from your block source
  3. Create a GitHub Release and attach the .wasm as a release asset
# Build your block as WASM
npx wafer-sdk-ts build -o myblock.wasm

# Tag and push
git tag v0.1.0
git push origin v0.1.0

# Then create a release on GitHub and upload the .wasm file
# Build your block as WASM
GOOS=wasip1 GOARCH=wasm go build -o myblock.wasm .

# Tag and push
git tag v0.1.0
git push origin v0.1.0

# Then create a release on GitHub and upload the .wasm file
# Build your block as WASM
cargo build --release --target wasm32-wasip1

# Tag and push
git tag v0.1.0
git push origin v0.1.0

# Then create a release on GitHub and upload the .wasm file

That's it. The registry will auto-index your package (name, description, versions) from GitHub the first time anyone requests it. Download counts are tracked automatically.

Publishing a Chain

Publishing a chain follows the same workflow — no registration, just GitHub:

  1. Push your chain definition to a public GitHub repository
  2. Write a .chain.json file describing your chain
  3. Create a GitHub Release and attach the .chain.json as a release asset

A chain definition is a JSON file that wires blocks together. For example:

{
    "root": {
        "block": "github.com/acme/auth-block@v1.0.0",
        "next": [
            { "match": "GET:/api/*", "block": "github.com/acme/api-handler@v2.0.0" },
            { "match": "*", "block": "github.com/acme/static-server@v1.0.0" }
        ]
    }
}

Name the file {repo}.chain.json (e.g. my-chain.chain.json) and attach it to your GitHub Release. The registry detects it automatically and indexes the package with type "chain".

Publishing an Interface

Interface packages are reusable JSON Schema type definitions that blocks can reference for their input/output contracts. Publishing follows the same GitHub workflow:

  1. Push your interface definition to a public GitHub repository
  2. Write a .interface.json file describing the shared schema
  3. Create a GitHub Release and attach the .interface.json as a release asset

An interface definition is a JSON file with a name, version, description, and a JSON Schema. For example:

{
    "name": "http-request",
    "version": "1.0.0",
    "description": "Standard HTTP request shape",
    "schema": {
        "type": "object",
        "properties": {
            "method": { "type": "string" },
            "path": { "type": "string" },
            "headers": { "type": "object", "additionalProperties": { "type": "string" } },
            "body": { "type": "string" }
        },
        "required": ["method", "path"]
    }
}

Name the file {repo}.interface.json (e.g. http-request.interface.json) and attach it to your GitHub Release. The registry detects it automatically and indexes the package with type "interface".

Interfaces are opt-in — blocks can still use inline JSON Schema for their input/output definitions. Interfaces are useful when multiple blocks need to share the same type contract.

Using an Interface

To use a published interface in your block, fetch it locally with the CLI:

wafer types add github.com/wafer-run/interfaces/http-request@v1.0.0

This downloads the .interface.json file to .wafer/interfaces/ and adds it to your wafer.json. Then reference it in your block.json:

{
    "actions": {
        "POST:/api/data": {
            "input": "github.com/wafer-run/interfaces/http-request@v1.0.0",
            "output": { "type": "object", "properties": { "ok": { "type": "boolean" } } }
        }
    }
}

When the input or output field is a string, it is treated as an interface reference. When it is an object, it is inline JSON Schema. Both coexist naturally — no migration needed.

Installing a Block

Reference a published block by name in your chain configuration with @version — the runtime will automatically download the .wasm file from the corresponding GitHub Release during startup:

{
    "root": {
        "block": "github.com/acme/auth-block@v1.0.0",
        "next": [
            { "match": "GET:/api/data", "block": "my-handler" }
        ]
    }
}

The version must match a GitHub Release tag (e.g. v1.0.0, v2.0.0-rc.1). The runtime expects a {repo}.wasm asset attached to the release — in this example, auth-block.wasm.

You can also omit the version to automatically resolve the latest release that has a .wasm asset:

{ "block": "github.com/acme/auth-block" }

The runtime queries the GitHub Releases API and picks the first release with a .wasm file attached. This is convenient for development, but pinning a version is recommended for production.

Installing a Chain

Reference a published chain the same way as a block, using @version. To download a chain definition instead of a .wasm file, use the ?type=chain query parameter on the download endpoint:

GET /registry/packages/github.com/acme/my-chain/download/v1.0.0?type=chain

This redirects to the .chain.json asset from the corresponding GitHub Release.

Installing an Interface

Download an interface definition using the ?type=interface query parameter:

GET /registry/packages/github.com/acme/http-request/download/v1.0.0?type=interface

This redirects to the .interface.json asset. The CLI wafer types add command does this automatically.

Browsing and Searching

  • Browse the registry at https://wafer.run/registry
  • Search via API: GET https://wafer.run/registry/search?q=auth

Only packages that have been looked up at least once appear in search results (just like Go's pkg.go.dev).

Registry API

All endpoints are at https://wafer.run:

  • GET /registry/search?q=term&type=block|chain|interface — search indexed packages. The optional type parameter filters results by checking if the type appears in the package's comma-separated type list. Omit to return all types.
  • GET /registry/packages/{owner}/{repo} — package details + versions (auto-indexes on first request). Response includes package_type field (comma-separated, e.g. "block", "chain", "interface", "block,interface").
  • GET /registry/packages/{owner}/{repo}/versions — version list from GitHub releases
  • GET /registry/packages/{owner}/{repo}/download/{version}?type=block|chain|interface — redirects to the GitHub release asset. Use type=chain for .chain.json, type=interface for .interface.json; defaults to block (.wasm).

Removing a Package

There is no delete API. If you remove your GitHub repository or delete a release, the versions will stop resolving. The package entry will remain in the index but with no available versions.

Blocking a Package

The registry includes an admin-only blocklist to prevent malicious or harmful packages from being indexed or returned in search results. Blocked packages cannot be auto-indexed from GitHub, and already-indexed packages that are later blocked will be filtered from search results and return 404 on direct lookup.

The blocklist is managed by wafer.run administrators — there is no public API to modify it. Admins can block individual packages (e.g. github.com/baduser/badrepo) or an entire owner's packages using a wildcard pattern (e.g. github.com/baduser/*).

Reporting a Bad Package

If you find a package that is malicious, contains malware, or otherwise violates the terms of use, please open a GitHub issue to request an admin review. Include the package name and a description of the problem, and we will investigate and block the package if necessary.

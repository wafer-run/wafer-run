CLI

The wafer CLI handles project scaffolding, building blocks to WebAssembly, running a local dev server with live reload, and publishing releases to GitHub.

Installation

npm install -g wafer-run

wafer init

Create a new project with a hello-world block, wafer.json, and package.json.

wafer init my-app
cd my-app
npm install

This creates:

  • wafer.json — block definitions and publish settings
  • package.json — npm project with wafer-sdk-ts as a dependency
  • blocks/hello/index.ts — a starter block that returns {"hello":"world"}
  • chains/main.json — the default chain configuration
  • tsconfig.json — TypeScript configuration for the project
  • .gitignore — ignores node_modules/, dist/, and *.wasm

wafer.json

The project config file generated by wafer init:

{
  "name": "my-app",
  "version": "0.1.0",
  "blocks": [
    { "name": "hello", "source": "blocks/hello" }
  ],
  "chains": [
    { "name": "main", "source": "chains/main.json" }
  ],
  "interfaces": [
    { "ref": "github.com/wafer-run/interfaces/http-request@v1.0.0",
      "local": ".wafer/interfaces/http-request@v1.0.0.interface.json" }
  ],
  "publish": {
    "repo": "wafer-run/my-app",
    "blocks": ["hello"]
  }
}

The interfaces array is optional and managed by wafer types add.

chains/main.json

Chain definitions live in the chains/ folder as separate JSON files:

{
  "id": "main",
  "root": { "block": "hello" }
}

wafer block

Add a new block to the project. Creates a block folder with an entry point and adds an entry to wafer.json.

wafer block greet

This creates blocks/greet/index.ts with a skeleton block class and adds it to wafer.json.

Language option

Use --lang to choose a different guest SDK:

wafer block greet --lang typescript   # default, creates blocks/greet/index.ts
wafer block greet --lang go           # creates blocks/greet/index.go
wafer block greet --lang rust         # creates blocks/greet/index.rs

wafer build

Compile all blocks defined in wafer.json to WebAssembly. Output goes to dist/.

wafer build

The compiler is selected automatically based on the entry point in each block folder:

  • .ts — TypeScript via npx wafer-sdk-ts build
  • .go — Go via GOOS=wasip1 GOARCH=wasm go build
  • .rs — Rust via cargo build --target wasm32-wasip1 --release

All blocks are built in parallel. If any block fails, the others still compile and the exit code is non-zero.

wafer dev

Build all blocks, start an HTTP dev server, and watch for changes.

wafer dev

The dev server loads each compiled .wasm block into a WaferRuntime, loads the chain files referenced in wafer.json, and listens on port 8080. Test it with:

curl http://localhost:8080
# {"hello":"world"}

Port option

wafer dev --port 3000

Live reload

When any file in blocks/ or wafer.json changes, the CLI rebuilds all blocks and restarts the runtime. Changes are debounced (300ms) so rapid saves don't cause multiple restarts.

The dev server requires wafer-run as a peer dependency. Install it with npm install wafer-run.

wafer publish

Build all blocks and create a GitHub release with the .wasm files as assets.

wafer publish

This reads publish.repo and publish.blocks from wafer.json, builds everything, and uploads the .wasm files to a GitHub release tagged v{version}.

Version bump

Use --bump to increment the version in wafer.json before publishing:

wafer publish --bump patch   # 0.1.0 → 0.1.1
wafer publish --bump minor   # 0.1.0 → 0.2.0
wafer publish --bump major   # 0.1.0 → 1.0.0

Authentication

Publishing uses the gh CLI if available (inherits your GitHub auth). If gh is not installed, it falls back to the GitHub REST API using a GITHUB_TOKEN environment variable.

Registry integration

Published .wasm assets are automatically discoverable by the package registry. The registry scans GitHub release assets for .wasm, .chain.json, and .interface.json files and makes them available for download.

wafer types

Manage interface type packages — reusable JSON Schema definitions that blocks can reference for their input/output contracts.

wafer types add

Fetch an interface package from the registry and add it to the project:

wafer types add github.com/wafer-run/interfaces/http-request@v1.0.0

This downloads the .interface.json file to .wafer/interfaces/ and adds an entry to the interfaces array in wafer.json. You can then reference it in your block.json:

{
  "actions": {
    "POST:/api/data": {
      "input": "github.com/wafer-run/interfaces/http-request@v1.0.0",
      "output": { "type": "object", "properties": { "ok": { "type": "boolean" } } }
    }
  }
}

When input or output is a string, it is treated as an interface reference. When it is an object, it is inline JSON Schema. Both work together — interfaces are opt-in for shared contracts.

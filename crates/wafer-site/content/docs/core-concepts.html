Core Concepts

Messages

Every request becomes a Message with three fields:

  • kind — the message type (e.g. "GET:/api/users")
  • data — the payload (raw bytes, typically JSON)
  • meta — a key-value map of metadata (headers, path params, action, etc.)

Helper methods on Message:

  • path() — read the request path from meta
  • action() — read the semantic action (retrieve, create, update, delete)
  • var(name) — read a path parameter (e.g. {id})
  • query(name) — read a query parameter
  • cont() — continue to the next block
  • respond(response) — short-circuit with a response

Blocks

A Block is any struct that implements the Block trait. Blocks are the fundamental unit of logic — they receive a message, do work, and return a result. Blocks can be native Rust code compiled into the binary, or WASM modules loaded at runtime.

Context

The Context is passed to every block's handle() method. It provides:

  • call_block(name, msg) — call another block by name (including infrastructure blocks like wafer/database, wafer/crypto, etc.)
  • config_get(key) — read a configuration value
  • is_cancelled() — check if the request has timed out or been cancelled

Use the wafer_run::clients module (or the Guest SDK typed clients for WASM blocks) for convenient typed access to infrastructure services like database, storage, crypto, etc.

Flows

Flows are trees of blocks. When a message arrives, it flows through the tree. Each node can match on the message (by HTTP method, path, etc.) and route to child blocks. Flows are defined in JSON and loaded at startup.

Interfaces

Interfaces are reusable JSON Schema type definitions that blocks can reference for their input/output contracts. Instead of repeating the same type definition across multiple blocks, define it once as an .interface.json package and reference it by name. Interfaces are opt-in — inline JSON Schema still works for simple cases. See Publishing an Interface and the CLI docs for wafer types add.

Actions

Each block returns one of four actions:

  • Continue — pass the message to the next block in the flow
  • Respond — send a response back to the caller immediately
  • Drop — silently discard the message
  • Error — signal an error (the flow's on_error policy decides what happens)
